### 第 7 页：步骤二&三：定义与研发 —— 🟢 文档与代码的敏捷闭环

**幻灯片标题：** 打破瀑布流：原型先行与文档反推的双向工作流

**高阶敏捷闭环（核心五步走）：**

1. **原型先行：** 直接使用 Cursor 生成原型，凭产品手感不断调试，直到觉得该有的功能逻辑都已经跑通。
2. **反推文档：** 基于跑通的原型界面与交互，利用 Gemini 反向撰写梳理出结构化的 PRD（包含你的 PM 身份、关注角度等长期记忆）。
3. **文档"盲测"：** 新开一个空白项目，只把写好的 PRD 喂给 Cursor，让它从 0 开始纯靠文档生成原型。
4. **纠偏迭代：** 对比新生成的原型与你最初想要的预期是否一致。如果不一致，说明 PRD 存在逻辑遗漏或描述不到位，反向倒逼修改 PRD。
5. **人工 Review 与正式评审：** AI 验证完毕后，最后由产品经理人工兜底 Review 确认全局逻辑。确认无误后，带着极其严密的 PRD 和在线原型去跟研发开需求评审会（所见即所得，彻底消除沟通代沟）。

**截图复刻利器：** 如果是在 MOI 现有页面上改版，第一步甚至可以直接截图老页面喂给 AI，在生成的代码上进行二次修改开发。

**最终黄金交付物：** 一份逻辑绝对严密、无懈可击的 PRD 文档 + 一个部署到 Vercel 的真实交互链接。

- **视觉建议：** 循环流程图排版，用箭头连接"建原型 ➔ 写PRD ➔ 盲测PRD ➔ 修改PRD ➔ 人工评审"，中间放置最终的交付物（文档+链接图标）。

### 第 8 页：研发协同实战案例：代码反向验证文档与深度思考

**幻灯片标题：** 消除盲区与 AI 时代研发范式的双重思考

**硬核实战（报销 Agent 案例）：**
- 新开空白项目，仅把最新 PRD 喂给 Cursor 看能否完美复刻。
- 收益：极大程度发现"纯靠脑子空想"想不到的边界漏洞。
- 能被机器执行的文档，才是好文档。

**深度思考一：研发链路的升维演进**
- 传统模式：需求 ➔ PRD ➔ Dev Design（技术设计） ➔ 代码 ➔ 测试用例 ➔ 产品
- AI 时代演进：未来极可能演变为 `(PRD + Dev Design + 测试用例) ➔ 代码`。
- 产品逻辑、技术架构与测试验收标准在极度严密的 Prompt 中走向三位一体的融合。
- **终极畅想：** 需求 ➔ 产品。在这个向终极形态过渡的阶段，PRD 的严密性（即能被机器无歧义执行）被提到了前所未有的高度。

**深度思考二："No-PRD"前沿趋势的冷思考与边界**
- **前沿动态：** Claude Code 团队已开始尝试"不写 PRD"，纯靠高保真交互原型交付。
- **批判性质疑：** 对于复杂系统，原型只能完美呈现 UI/UX 的"显性逻辑"，但极难承载"隐性逻辑"（如底层算法规则、并发处理、异常流转限制等）。
- **协作新范式：** 划定清晰边界。原型承载"所见即所得"的体验层，PRD 退阶为"纯享版"的后台规则库。（凡是能在原型里演示明白的就不在文档废话；PRD 专心沉淀原型画不出来的规则底座）。

- **视觉建议：** 建议使用"冰山模型"的图示。海面上是"可见的交互原型（UI/UX/主流向）"，海面下是庞大隐蔽的"隐形 PRD 逻辑（状态机、异常分支、数据清洗规则）"。

```
# 角色设定
你现在是一位拥有10年经验的资深产品架构师，同时也是一位以"找漏洞"著称、极度严苛的 QA 测试专家。你的逻辑极其严密，擅长从看似完美的流程中发现致命的边界问题。

# 背景信息
我们正在为【{产品信息}】开发一个全新的功能模块。我深知一个 Solid（立得住）的产品功能 = 清晰的主干用户旅程 + 对无数边界/异常问题的解答。

# 输入信息
以下是我初步梳理出的该功能的「核心用户旅程（Happy Path）」及基本规则：
【{在此处描述该功能的主干流程，例如：用户点击入口 -> 系统校验 -> 弹出弹窗 -> 用户提交 -> 系统处理反馈}】

# 具体任务
请你仔细阅读上述主干流程，发挥你作为"提问器"的作用，对我进行"疯狂提问"。
你需要穷举找出这个旅程中所有潜在的逻辑漏洞、缺失的异常流（Unhappy Path）、极端场景（Edge Cases）、并发冲突以及技术层面的边界限制条件。

# 输出要求
1. 只提问，不解答：我不需要你给出解决方案，我只需要你提出最犀利、最能击中痛点的"灵魂拷问"。
2. 结构化输出：请将你的问题归类输出为一份【高密度 Checklist】，分类建议如下（可根据实际情况调整）：
   - 前置条件与权限校验
   - 异常操作与中断流转（如：弱网、中途退出、重复点击）
   - 数据状态与并发限制（如：多人同时操作同一对象）
   - 极端末端场景与边界值
   - 与现有系统/模块的耦合影响
3. 语气：专业、犀利、直击要害。
```

**立项黄金法则：** 主流程清晰 + AI 抛出的"一堆问题"全解答 = 坚如磐石的核心功能。

### 第 10 页：步骤二&三：定义与研发 —— 🔴 AI 的盲区

**幻灯片标题：** 审美与用户体验：AI 只是"毛坯房"建造者

- **核心局限一：大模型对"美"没有直觉。**
  - 缺乏人类的同理心、留白感知和视觉层级判断。
  - 原型往往"能用"但"不好用"（如按钮位置别扭、微交互生硬）。
- **核心局限二：无法量化"不好用"的脑科学难题。**
  - 当所有用户都觉得产品"不好用"时，这种体验是一种极其复杂、基于人类神经学与心理学的综合反馈。
  - **不可量化困境：** 目前的 AI 根本不具备人类的体验判断能力，也无法将这种"不爽"的情绪量化成模型能够理解的参数并进行自动优化。（要突破这一层，可能需要脑科学与人工智能结合的底层技术革命）。
- **应对姿势：** 明确分工。AI 负责超低成本拉齐功能逻辑（造毛坯房），但最终的"精装修"（UX/UI 的打磨、情感化传达与体验升华）必须依赖 PM 的共情能力和专业设计师的介入。

- **视觉建议：** 对比图形式。左图是"功能堆砌的粗糙界面（毛坯房）"，右图是"精雕细琢的优秀 UX（精装修）"。可以辅以一个"大脑神经元网络"的图标来强调体验量化难题。

### 第 14 页：产品人生存法则 (Survival Rules)

**幻灯片标题：** 产品人的五条生存法则

1. **你是决策者，AI 是副驾驶：** 对 AI 产出的逻辑自洽性、数据合理性、用户场景还原度保持审辨力，为最终决策全程负责，坚决不做甩手掌柜。
2. **掌握 Prompt 公式：** 角色设定 + 背景信息 + 具体任务 + 输出要求。利用大模型记忆功能固化前两项；越充分的上下文，越高质量的输出。
3. **重复三次，就该 AI 化：** 建立流程初期可能更费时费力，但一旦跑通，长期收益远超前期投入。把重复性工作交给 AI，把稀缺注意力留给真正需要人判断的事。
4. **从执行者到审查者：** AI 能快速生成原型和需求文档后，PM 的核心价值转移到审查质量——看得出交互逻辑是否自洽，需求描述是否覆盖边界场景，这才是不可替代的能力。
5. **跨界融合：产品与研发界限模糊：** PM 需学会 review 代码，验证 vibe coding 质量；研发需转变为用户/需求视角，从市场与商业角度思考。主动拥抱对方领域的人将成为 AI 时代最稀缺的复合型人才。

- **视觉建议：** 两列布局，左列放前三条，右列放新增两条（构建者/审查者、跨界融合）。
